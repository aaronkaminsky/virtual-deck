# Phase 999.11: Pile Drop Dialog UX Improvements — Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Keyboard UX improvements to the existing pile insert-position dialog (Top / Bottom / Random in `BoardDragLayer.tsx`). Two behavior changes:
1. Escape key cancels the drop — card stays at its origin on the server (never moved)
2. Enter key confirms Top — the default position

**In scope:**
- Escape cancels: clear `pendingMove` without calling `sendPendingMove`, so the card stays at its server-side origin
- Click-outside cancels: same behavior as Escape (any dismissal = cancel)
- Enter confirms Top: achieved by auto-focusing the Top button when the dialog opens
- Top button gets primary (filled) visual style to signal it as the default

**Out of scope:**
- New insert positions or dialog layout changes
- Changing which positions are available (Top / Bottom / Random remain)
- Any changes to how the dialog is triggered (drop interception in `handleDragEnd` is unchanged)

</domain>

<decisions>
## Implementation Decisions

### Escape behavior
- **D-01:** Escape key **cancels** the move. `pendingMove` is cleared without calling `sendPendingMove`. Card remains at its server-side origin (the `pendingMove` pattern means the server never received the MOVE_CARD action, so no revert is needed).
- **D-02:** The `@base-ui/react/dialog` fires `onOpenChange(false)` for both Escape and click-outside. Use the `onEscapeKeyDown` prop (or equivalent in @base-ui) to intercept Escape separately from click-outside, OR set `onOpenChange` to always cancel (since D-03 makes click-outside cancel too).

### Click-outside behavior
- **D-03:** Click-outside also **cancels** (same as Escape). Any dismissal = cancel. The only way to commit a move is to click Top, Bottom, or Random. This replaces the prior D-03 from phase 999.2 (which confirmed Top on any dismiss).

### Enter key
- **D-04:** Enter confirms Top via **auto-focus**: the Top button receives focus when the dialog opens. Enter naturally activates the focused button — no bespoke keydown handler needed. Tab cycles through Bottom → Random → Top.

### Top button visual
- **D-05:** Top button gets **primary button style** (filled/solid, using the existing `Button` component with `variant="default"` or equivalent). Bottom and Random remain as bordered/ghost buttons. Visual hierarchy signals Top as the default.

### Claude's Discretion
- Exact @base-ui Dialog prop for intercepting Escape vs. onOpenChange (check @base-ui docs — may be `closeOnEscape={false}` + manual handler, or `onEscapeKeyDown`)
- Whether `autoFocus` is on the button element itself or via a Dialog prop
- Button component: can use the existing `Button` from `src/components/ui/button.tsx` with `variant="default"` for Top and `variant="outline"` (or ghost) for Bottom and Random

</decisions>

- `onOpenChange={(open) => { if (!open) sendPendingMove('top'); }}` is the existing dismiss handler — replace with cancel logic
- `pendingMove !== null` controls dialog open state — clearing it without sending = cancel

### Integration Points
- No server changes needed — `pendingMove` is a client-side intercept; the server only acts when `sendPendingMove` fires
- `autoFocus` on the Top button or via dialog `initialFocus` prop enables keyboard Enter path
