## Warnings

### WR-01: Unsafe cast of `event.active.data.current` can crash on malformed drag sources

**File:** `src/components/BoardDragLayer.tsx:76`

**Issue:** `handleDragStart` casts `event.active.data.current` directly to `{ card: Card; fromZone: string; fromId: string }` with no runtime check. If any droppable that doesn't carry this shape ever fires a drag-start (e.g., a future drag source added elsewhere in the tree), `data.card` will be `undefined`, causing `setActiveCard(undefined)` — TypeScript won't catch this because the cast bypasses the type system. The same value flows through `dragDataRef.current` and is later non-null-asserted at lines 93 and 103.

**Fix:**
```typescript
function handleDragStart(event: DragStartEvent) {
  const data = event.active.data.current as { card?: Card; fromZone?: string; fromId?: string } | undefined;
  if (!data?.card || !data.fromZone || !data.fromId) return; // guard against unexpected drag sources
  dragDataRef.current = data as { card: Card; fromZone: string; fromId: string };
  setActiveCard(data.card);
  setDragging(true);
}
```

### WR-02: `sendPendingMove` silently no-ops when `pendingMove` is null; no indication to the caller

**File:** `src/components/BoardDragLayer.tsx:61-73`

**Issue:** The early return on line 62 (`if (!pendingMove) return;`) is correct as a guard, but the buttons calling `sendPendingMove` are rendered only when `pendingMove !== null` (the dialog is `open={pendingMove !== null}`). The guard therefore masks a logic gap: if `sendPendingMove` is ever called outside that dialog (e.g., via keyboard shortcut or future refactor), the move is silently dropped with no error, no toast, and no state change. The action is a destructive no-op — the card never reaches the pile.

**Fix:** Make the invariant explicit so it fails loudly during development:

```typescript
function sendPendingMove(insertPosition: 'top' | 'bottom' | 'random') {
  if (!pendingMove) {
    if (process.env.NODE_ENV === 'development') {
      console.error('sendPendingMove called with no pendingMove — this is a bug');
    }
    return;
  }
  // ... rest unchanged
}
```

## Info

### IN-01: `[a]:hover:bg-primary/80` selector in `button.tsx` is an invalid Tailwind arbitrary variant and will never apply

**File:** `src/components/ui/button.tsx:12`

**Issue:** The `default` variant class string contains `[a]:hover:bg-primary/80`. The `[a]:` prefix is intended as a Tailwind arbitrary variant to scope the hover to anchor-tag contexts, but `[a]` is not a valid CSS selector in Tailwind's arbitrary variant syntax (which expects a CSS pseudo-class or a `&` selector pattern, e.g., `[&_a]:hover:...` or `a:hover:...`). The rule will never be emitted by Tailwind, so `<a>` elements styled as buttons will not get a hover background. This is a latent dead rule, not a crash.

**Fix:** If the intent is "apply hover background when the button is used as an anchor-tag link," use:
```
hover:bg-primary/80
```
If it should only apply when inside an `<a>`, use the correct arbitrary-variant form:
```
[&_a]:hover:bg-primary/80
```
Most likely the intent is a plain hover for the default variant, in which case simply use `hover:bg-primary/80`.

### IN-02: Magic number `defaultDropAnimation.duration + 50` repeated twice without a named constant

**File:** `src/components/BoardDragLayer.tsx:129`, `src/components/BoardDragLayer.tsx:139`

**Issue:** The `50` millisecond buffer added to the animation duration appears in two separate `setTimeout` calls. The intent (wait slightly longer than the animation to avoid a visual flash) is clear from context, but the value has no label and any future animation-duration change requires updating both sites.

**Fix:**
```typescript
const SNAP_BACK_BUFFER_MS = 50;

// then at both call sites:
setTimeout(() => setActiveCard(null), defaultDropAnimation.duration + SNAP_BACK_BUFFER_MS);
```
