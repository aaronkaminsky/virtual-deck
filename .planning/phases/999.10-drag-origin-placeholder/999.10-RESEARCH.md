# Phase 999.10: Drag Origin Placeholder — Research

**Researched:** 2026-04-14
**Domain:** React drag-and-drop visual feedback (dnd-kit, inline conditional rendering)
**Confidence:** HIGH

## Summary

This phase is a pure client-side visual change: replace the invisible card (`opacity: 0`) that dnd-kit leaves in the DOM during a drag with a visible dashed outline placeholder. No server interaction, no new libraries, no new state management. All the building blocks already exist in the codebase.

The two change points are `DraggableCard.tsx` (pile cards, uses `useDraggable`) and `SortableHandCard` inside `HandZone.tsx` (hand cards, uses `useSortable`). In both components, `isDragging` is already available from the hook return value. The placeholder is a dashed-border `div` rendered in place of `CardFace`/`CardBack` when `isDragging` is true — the outer wrapper stays in the DOM unchanged, which is what dnd-kit requires for position tracking.

Placeholder teardown on cancel must wait until the snap-back animation finishes. `defaultDropAnimation.duration` is 250 ms [VERIFIED: node runtime]. The existing `handleDragCancel` in `BoardDragLayer` already uses `defaultDropAnimation.duration + 50` (300 ms total) for overlay teardown. The placeholder lifespan must match this same window — it disappears when `isDragging` becomes false after the animation settles, which happens naturally because dnd-kit sets `isDragging` back to false only after `onDragEnd`/`onDragCancel` fires.

**Primary recommendation:** Inline conditional in both `DraggableCard` and `SortableHandCard`: when `isDragging`, render a dashed-border div using `border-dashed border-border rounded-lg w-[80px] h-[112px]` in place of the card content. No new component needed.

### D-03: Why SortableHandCard Gets Placeholder for All isDragging States

The CONTEXT.md (code_context section) notes: "The `isDragging` flag is true in both [hand-to-hand reorder AND cross-zone hand drags]. The decision (D-03) to skip placeholder for hand-to-hand reorder means the placeholder should apply to all `isDragging` states in `SortableHandCard` — the sortable shift for reorder is provided by dnd-kit and won't conflict visually since the origin card is what shifts, not a separate placeholder."

In practice: when `SortableHandCard` is reordering within the same hand, the `isDragging` card becomes the placeholder at its origin position. The other cards in the hand shift around it via dnd-kit's `transform + transition`. This is acceptable — the dashed outline at the origin, combined with adjacent cards shifting, reads correctly to the user. No special detection of hand-to-hand vs cross-zone is needed inside `SortableHandCard` itself.

### Cancel / Snap-Back Timing

```
defaultDropAnimation = { duration: 250, easing: 'ease' }
// [VERIFIED: node -e "require('@dnd-kit/core')" runtime check]

handleDragCancel (BoardDragLayer.tsx line 110):
  setTimeout(() => setActiveCard(null), 250 + 50)  // 300ms total
```

The placeholder is controlled by `isDragging` from dnd-kit. On drag cancel:
1. `onDragCancel` fires immediately → `isDragging` stays `true` during the snap-back animation
2. The overlay (`CardOverlay`) animates back to the placeholder position over 250ms
3. After the snap-back completes, dnd-kit sets `isDragging = false` → placeholder disappears, card content reappears
4. `setActiveCard(null)` fires at 300ms, clearing the overlay

This means placeholder teardown is automatic — `isDragging` goes false when dnd-kit finishes the cancel animation. No additional timeout or state needed in `DraggableCard` or `SortableHandCard`.

**Important:** On successful drop, `onDragEnd` fires and dnd-kit sets `isDragging = false` immediately (before `activeCard` is cleared). The game state update from the server will re-render with the card gone. The placeholder will flicker off at `isDragging = false`, then the slot becomes empty or occupied by the new card — this is the correct behavior.

## Common Pitfalls

### Pitfall 1: Sizing Mismatch Between Pile and Hand Placeholders
**What goes wrong:** Using 80×112px for the hand placeholder inflates the hand strip height or breaks the flex layout.
**Why it happens:** `DraggableCard` sits inside PileZone which is already sized at 80×112. `SortableHandCard` sits in a flex row sized for 63×88 card content.
**How to avoid:** Use 80×112 for `DraggableCard` placeholder, 63×88 for `SortableHandCard` placeholder.
**Warning signs:** Hand strip height changes when a card is dragged.

### Pitfall 2: Leaving opacity in the style Object
**What goes wrong:** If `opacity` is left in the style and also conditional content is used, the dragged card becomes double-hidden (opacity 0 + content replaced), causing a flash when `isDragging` goes false.
**Why it happens:** Both the old `opacity` and the new content branch control visibility simultaneously.
**How to avoid:** Remove `opacity: isDragging ? 0 : 1` from the style object entirely when switching to content-branch approach.

### Pitfall 3: Removing the wrapper's ref or listeners
**What goes wrong:** If the conditional rendering accidentally puts the dnd-kit `ref` on the placeholder instead of the outer wrapper, dnd-kit loses position tracking.
**Why it happens:** Developer restructures JSX during the refactor.
**How to avoid:** The outer `div` with `ref={setNodeRef}` must stay unchanged; only the children change.

### Pitfall 4: Forgetting `transition` in SortableHandCard style
**What goes wrong:** Other cards in the hand stop animating during sort after the refactor.
**Why it happens:** `SortableHandCard` includes `transition` in its style object (needed for sortable shifts). If the style object is accidentally dropped during the refactor, adjacent card animations break.
**How to avoid:** Keep `transition` in style for `SortableHandCard`; only remove `opacity` from it.

### Pitfall 5: Placeholder Appearing During Successful Drop
**What goes wrong:** After a successful drop, there's a visible flash of the dashed placeholder before the game state update removes the card from its origin.
**Why it happens:** `isDragging` goes false immediately on `onDragEnd` (successful path), but the server round-trip hasn't updated the card list yet.
**Severity:** LOW — the round-trip is fast (PartyKit WebSocket) and the card was already sent to its destination. Players see the overlay animate to target and then the origin clears. This is the expected behavior; the placeholder flash (if any) is sub-frame.
**How to avoid:** No action needed — this is the intended behavior per D-05.

## Open Questions

1. **Hand placeholder rounded corner radius**
   - What we know: `CardFace`/`CardBack` use `rounded-md` (line 28 CardFace, line 12 CardBack). PileZone container uses `rounded-lg`.
   - What's unclear: Should the hand placeholder match the card content radius (`rounded-md`) or use `rounded-lg` for consistency?
   - Recommendation: Use `rounded-md` for the hand placeholder to match `CardFace`/`CardBack` exactly, since the placeholder stands in for the card face.

2. **Border width for hand placeholder**
   - What we know: PileZone uses Tailwind's default `border` (1px). The placeholder in hand is 63×88px — at 1px dashed it may be hard to see.
   - What's unclear: Should hand placeholder use `border-2` to be more visible at smaller size?
   - Recommendation: Start with `border` (1px, consistent with empty pile). Adjust to `border-2` only if visual testing shows it's too subtle.
