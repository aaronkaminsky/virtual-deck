<objective>
Replace the global `closestCenter` collision algorithm in `DndContext` with a custom collision function that uses `pointerWithin` for zone-level droppables (`'hand'` and `opponent-hand-*`) while preserving `closestCenter` for sortable card-to-card interactions.

Purpose: The hand strip (`HandZone`) and opponent hand (`OpponentHand`) attach their droppable ref to full-width flex containers. `closestCenter` registers a hit as soon as the dragged item's center is nearest that container's center — which can be far from the visible card area. `pointerWithin` only registers a hit when the pointer is physically inside the droppable's rect, so the drop zone boundary matches what the user sees. Pile drops are unaffected (pile containers are correctly sized 80×112px). Sortable card-to-card reordering is unaffected (card IDs are not zone-level droppables).

Output: One modified file (`BoardDragLayer.tsx`) with a `customCollision` function replacing the `closestCenter` import in the `collisionDetection` prop.

```tsx
/**
 * Custom collision detection for the board:
 *
 * - Zone-level droppables ('hand' and 'opponent-hand-*') use pointerWithin so that a
 *   drop only registers when the pointer is physically inside the droppable element's
 *   rendered rect. These containers span the full viewport width; closestCenter would
 *   fire too eagerly and prevent the user from triggering a missed-drop snap-back.
 *
 * - All other droppables (pile zones sized 80x112, individual sortable card IDs) use
 *   closestCenter, which is the correct algorithm for pile targets and for the
 *   @dnd-kit/sortable card-to-card reorder interaction inside HandZone.
 */
const customCollision: CollisionDetection = (args) => {
  // Separate zone-level droppables from everything else.
  const zoneIds = new Set(['hand']);
  const zoneContainers = args.droppableContainers.filter(
    (c) => zoneIds.has(String(c.id)) || String(c.id).startsWith('opponent-hand-')
  );
  const otherContainers = args.droppableContainers.filter(
    (c) => !zoneIds.has(String(c.id)) && !String(c.id).startsWith('opponent-hand-')
  );

  // For zone-level droppables, only register a hit when the pointer is inside the rect.
  const zoneCollisions = pointerWithin({ ...args, droppableContainers: zoneContainers });

  // For all other droppables (piles, sortable card IDs), use closestCenter.
  const otherCollisions = closestCenter({ ...args, droppableContainers: otherContainers });

  // Prefer a pointer-within zone hit if one exists; fall back to closest-center results.
  // This means: if the pointer is inside the hand strip, that zone wins over a nearby pile.
  // If the pointer is not inside any zone, pile/card collisions are returned normally.
  return zoneCollisions.length > 0 ? zoneCollisions : otherCollisions;
};
```

**Why this approach:**

`pointerWithin` checks whether `args.pointerCoordinates` is inside each droppable's `DOMRect`. For `HandZone`, the 128px strip occupies only the bottom portion of the screen — the pointer must be inside that strip for a hit to register. For `OpponentHand`, the droppable is the rendered component container (which dynamically shows `min-h-[44px] min-w-[80px]` during drag via the existing `dragIsActive` conditional class), so the hit area is the visible opponent badge area.

The fallback to `closestCenter` for `otherContainers` ensures that:
1. Pile drops remain unchanged (pile zones are 80×112px, already correctly sized).
2. Hand-to-hand sortable reordering still works: sortable card IDs are processed through `closestCenter`, which is what `@dnd-kit/sortable` expects internally for the horizontal strategy.
