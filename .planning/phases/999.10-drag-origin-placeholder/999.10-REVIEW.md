## Summary

This phase introduced two features: (1) a drag-origin placeholder (dashed outline at the card's slot while dragging) threaded via `draggingCardId` through `BoardDragLayer` → `BoardView` → `PileZone`/`HandZone` → `SortableHandCard`, and (2) a split of the old `isHandToHand` flag into `isHandReorder` (drop over a specific card) and `isHandMissed` (drop over the bare `hand` droppable zone), with `isHandMissed` correctly routing to the snap-back animation path.

The logic for both features is structurally correct. The `isHandMissed` path in `useDndMonitor` correctly no-ops due to the `overIdx !== -1` guard. The placeholder rendering in `SortableHandCard` and `PileZone` is correct.

Two issues were found: one critical (unsafe non-null assertion in the `isPassCard` branch of `handleDragEnd`) and one warning (permanent `didDragRef` suppression after a drag that ends without triggering a click event).

## Critical Issues

### CR-01: Non-null assertion on `dragDataRef.current` in `isPassCard` branch without null guard

**File:** `src/components/BoardDragLayer.tsx:58-65`

**Issue:** `isPassCard` is computed solely from `overData?.toZone === 'opponent-hand'` (line 58) — it does not check `dragDataRef.current !== null`. The `isSuccess` condition at line 59 does explicitly include `dragDataRef.current` in its guard, but `isPassCard` does not. At line 65, `dragDataRef.current!` is used with a non-null assertion. If a `dragend` event fires without a preceding `dragstart` having set `dragDataRef.current` (a possible edge case under rapid pointer events or DndContext re-mounts), this throws a runtime `TypeError: Cannot destructure property 'card' of null`.

**Fix:**
```tsx
// BoardDragLayer.tsx, handleDragEnd — add dragDataRef.current guard to isPassCard
const isPassCard = !!(overData?.toZone === 'opponent-hand' && dragDataRef.current);
```
This mirrors the same defensive pattern already used in `isSuccess` at line 59. No other changes needed — the destructure at line 65 (`dragDataRef.current!`) is then safe.

## Warnings

### WR-01: `didDragRef` in `DraggableCard` stays `true` permanently after drag-without-click, suppressing the next legitimate flip

**File:** `src/components/DraggableCard.tsx:21-31`

**Issue:** `didDragRef.current` is set to `true` when `isDragging` becomes true (line 23). It is reset to `false` only inside `handleClick` (line 28). If a drag ends on a different drop target (no `click` event fires on this element), `didDragRef.current` remains `true` for the lifetime of that component instance. The very next single-click on the same card — a legitimate flip intent — is silently swallowed.

In practice this self-heals when the card moves zones (component unmounts and remounts), but if the drag is cancelled or snaps back to the same pile slot, the card is left in a state where one flip is permanently suppressed until another drag-and-drop cycle completes.

**Fix:** Reset the ref on drag end (when `isDragging` transitions from `true` to `false`) rather than only in the click handler.

```tsx
const prevIsDragging = useRef(false);
useEffect(() => {
  if (prevIsDragging.current && !isDragging) {
    // Drag just ended — arm the suppression; a click within the same event
    // delivery will clear it. Use a timeout so the click handler fires first.
    setTimeout(() => { didDragRef.current = false; }, 300);
  }
  if (isDragging) didDragRef.current = true;
  prevIsDragging.current = isDragging;
}, [isDragging]);
```

Alternatively, reset directly in `handleDragCancel`/`handleDragEnd` in `BoardDragLayer` via a callback prop — whichever pattern is consistent with the rest of the codebase.

## Info

### IN-01: `topCard ?? {}` uses an unsafe empty-object fallback for an `'id' in …` check

**File:** `src/components/PileZone.tsx:54`

**Issue:** The expression `'id' in (topCard ?? {})` uses a plain `{}` as a fallback when `topCard` is `null`. This works because `'id' in {}` returns `false`, but TypeScript types `{}` as `object`, not as any card shape. The intent (check whether `topCard` has an `id` property) is correct but the pattern is opaque and would break if the fallback value ever changed.

**Fix:** Use an explicit null guard instead:
```tsx
{topCard && 'id' in topCard
  ? <DraggableCard card={topCard as Card} fromZone="pile" fromId={pile.id} onFlip={handleFlipCard} />
  : topCard && <CardBack />}
```
