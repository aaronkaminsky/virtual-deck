## Summary

Reviewed the three drag-and-drop components in scope: `BoardDragLayer` (drag orchestration, overlay, pile-insert dialog), `DraggableCard` (draggable primitive with flip-suppression), and `HandZone` (sortable hand strip with reorder monitor). The implementation is structurally sound — the `isPassCard` guard from the prior code-review fix is in place, and the `didDragRef` reset-on-drag-end pattern is implemented correctly. Two new warnings were found: an unsafe cast on `event.active.data.current` that will throw if any draggable lacks card data, and a stale `dropSuccessRef` that can cause the overlay's snap-back animation to be skipped on the drag following a successful drop. Two info items cover `setTimeout` leak on unmount and a silent no-op in hand reordering.

## Warnings

### WR-01: Unsafe cast of `event.active.data.current` in `handleDragStart` — will throw if data is absent

**File:** `src/components/BoardDragLayer.tsx:82-84`

**Issue:** `event.active.data.current` is cast directly to `{ card: Card; fromZone: string; fromId: string }` with no null guard. If any draggable registered inside the `DndContext` does not provide a `data` object (a future component, a mis-configured `useDraggable`, or a test harness), `data.card` throws `TypeError: Cannot read properties of undefined`. `dragDataRef.current` is set unconditionally on line 83, propagating the bad value into all downstream logic in `handleDragEnd`.

**Fix:**
```typescript
function handleDragStart(event: DragStartEvent) {
  const data = event.active.data.current as { card: Card; fromZone: string; fromId: string } | undefined;
  if (!data?.card) return; // guard against draggables without card data
  dragDataRef.current = data;
  setActiveCard(data.card);
  setDragging(true);
}
```

---

### WR-02: `dropSuccessRef` is not reset at drag-start — stale `true` disables snap-back animation on the following drag

**File:** `src/components/BoardDragLayer.tsx:94` and `158`

**Issue:** `dropSuccessRef.current` is written during `handleDragEnd` (line 94) and reset to `false` during `handleDragCancel` (line 142), but is never reset in `handleDragStart`. `DragOverlay` reads it at render time (line 158): when it is `true`, `dropAnimation` is set to `null`, disabling snap-back. If drag N is a successful drop, `dropSuccessRef.current` stays `true`. When the user starts drag N+1 and misses a drop target, the overlay uses `null` (no animation) instead of `defaultDropAnimation`, so the card jumps back visually instead of animating.

**Fix:** Reset the ref at the start of each drag:
```typescript
function handleDragStart(event: DragStartEvent) {
  dropSuccessRef.current = false; // ensure fresh drag starts with snap-back enabled
  const data = event.active.data.current as { card: Card; fromZone: string; fromId: string };
  dragDataRef.current = data;
  setActiveCard(data.card);
  setDragging(true);
}
```

## Info

### IN-01: `setTimeout` callbacks cannot be cancelled if component unmounts mid-drag

**File:** `src/components/BoardDragLayer.tsx:135` and `145`

**Issue:** Two `setTimeout` calls schedule `setActiveCard(null)` after a delay to allow the snap-back animation to complete. If `BoardDragLayer` unmounts before the timer fires (e.g., the user navigates away from the room during the animation), the callbacks run on an unmounted component. React 18 silently no-ops the state setter, so there is no crash, but the timers cannot be cancelled and create a minor leak.

**Fix:** Track the timer ID in a ref and cancel it on cleanup:
```typescript
const snapBackTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

useEffect(() => {
  return () => {
    if (snapBackTimerRef.current !== null) clearTimeout(snapBackTimerRef.current);
  };
}, []);

// Replace both setTimeout calls:
snapBackTimerRef.current = setTimeout(() => setActiveCard(null), defaultDropAnimation.duration + 50);
```

### IN-02: Hand reorder silently no-ops when a card is dropped onto empty zone space rather than a sibling card

**File:** `src/components/HandZone.tsx:76-77`

**Issue:** In `useDndMonitor`'s `onDragEnd`, `overIdx` is computed by searching `cards` for `over.id`. When the drop target is the `'hand'` zone droppable itself (not a specific card), `over.id` is `'hand'`, which is not a card ID — `overIdx` is `-1` and the guard `overIdx !== -1` suppresses the `REORDER_HAND` action. A user dragging a card to an empty area at the end of the hand strip sees the card snap back with no reorder applied.

This may be intentional (require targeting a specific slot to reorder). If the intent is to also allow appending by dropping in empty space, treat `-1` as the last position:
```typescript
const resolvedOverIdx = overIdx !== -1 ? overIdx : cards.length - 1;
if (activeIdx !== -1 && activeIdx !== resolvedOverIdx) {
  const reordered = arrayMove(cards, activeIdx, resolvedOverIdx);
  sendAction({ type: 'REORDER_HAND', orderedCardIds: reordered.map(c => c.id) });
}
```
