<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → PartyKit `onMessage` | Untrusted JSON action (MOVE_CARD, SET_PILE_FACE, etc.) crosses here. Server already validates fromZone/fromId against senderToken for hand sources. |
| client → PartyKit `onConnect` URL | Untrusted URL query parameters (`player`, `name`). Server already truncates player to 64 chars and strips HTML-special characters from name. |
| durable-object storage → onStart | Persisted gameState from previous deploy crosses here. Must tolerate old-shape records (missing region/ownerId). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Tampering | `onConnect` spread zone creation | mitigate | Zone id is server-computed as `` `spread-${playerToken}` `` where `playerToken = rawToken.slice(0, 64)`. Idempotency check uses `pile.id === spreadZoneId`, never a client-supplied id or `pile.name`. Prevents zone-id injection and duplicate-zone creation via display-name fuzzing. |
| T-14-02 | Tampering / Access Control bypass | `MOVE_CARD` toId pointing at another player's personal spread zone | accept | Per D-16 (no rule enforcement — any player may drag to any zone). Server still validates the destination pile exists via `piles.find(p => p.id === toId)`; nonexistent spread ids return PILE_NOT_FOUND (existing behavior). No privacy leak: spread zone cards are fully public. |
| T-14-03 | Denial of Service | Reconnect spam inflating `piles[]` with duplicate spread zones | mitigate | Idempotency check in `onConnect` (`piles.some(p => p.id === spreadZoneId)`) prevents duplicates. 4-player room cap (existing `players.length >= 4` guard) bounds distinct spread zones to 4 + 1 communal. |
| T-14-04 | Information Disclosure | `viewFor` masking logic leaks spread zone card identity | mitigate | Spread zones default to `faceUp: true` — all cards pass the `card.faceUp || isTop` check and remain visible. Face-down spread zones intentionally mask non-top cards (same as piles); this is the documented behavior, not a leak. No privacy regression vs v1.1 pile masking. |
| T-14-05 | Tampering | Stale persisted gameState from prior deploy lacks `region`/`ownerId` fields | mitigate | `onStart` migration runs field defaults (`region = "pile"`, `ownerId = null`) on every existing pile before any logic depends on those fields. Communal-zone existence check runs AFTER field migration, so a newly-pushed communal zone still gets both fields set explicitly. |
</threat_model>
