## Files Created/Modified

- `src/components/OpponentHand.tsx` - Added useDndContext, dragIsFromHand boolean, three-state border, min size during drag, "Drop to pass" label; then corrected affordance trigger to fire for any drag (not just fromZone=hand check that was too narrow)
- `tests/undoMove.test.ts` - Added "undo after PASS_CARD restores both sender and recipient hands" test case
- `src/components/PileCard.tsx` - Revealed top card's id via droppable data even for face-down piles so drag overlay and flip can target it
- `src/components/Pile.tsx` - Blocked click-to-flip on face-down cards inside a face-down pile (was unintentionally allowing flip via click when pile orientation should own face state)
- `src/components/BoardDragLayer.tsx` - Fixed two issues: (a) preserved card's faceUp=false during drag overlay render, (b) enabled pile-to-opponent-hand pass by relaxing fromZone check

## Decisions Made

- `useDndContext` is the right hook for detecting drag state in deeply nested components — avoids threading drag state as props through the tree
- Drag affordance fires for ANY active drag (not just fromZone === 'hand') after verification revealed that the fromZone check was too restrictive in practice; the droppable zone should always be visible when dragging anything that could be passed

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Face-down pile top card not exposing id for drag**
- **Found during:** Task 2 (manual verification — drag from pile to opponent hand)
- **Issue:** PileCard was not including `card.id` in droppable data for face-down orientation, so the drag layer could not identify the card being dragged
- **Fix:** Always include card.id in PileCard droppable data regardless of face orientation
- **Files modified:** src/components/PileCard.tsx
- **Verification:** Drag from face-down pile to opponent hand dispatched PASS_CARD correctly after fix
- **Committed in:** abfc9b2

**2. [Rule 1 - Bug] Click-to-flip triggering on face-down cards in face-down piles**
- **Found during:** Task 2 (manual verification — clicking pile cards)
- **Issue:** Cards inside a face-down pile were individually flippable via click, which conflicts with pile-level face orientation ownership
- **Fix:** Blocked click-to-flip handler on face-down cards within face-down piles (pile orientation takes precedence)
- **Files modified:** src/components/Pile.tsx
- **Verification:** Cards in face-down piles no longer flip individually on click
- **Committed in:** 41eb252

**3. [Rule 1 - Bug] Drag overlay showed card face-up even for face-down cards; pile-to-opponent pass blocked**
- **Found during:** Task 2 (manual verification — drag visual and drag-from-pile-to-opponent)
- **Issue:** BoardDragLayer was not preserving faceUp=false during overlay render; also an overly strict fromZone check blocked pile-to-opponent-hand passes
- **Fix:** Threaded faceUp from active drag data into overlay card render; relaxed fromZone check to allow pile source
- **Files modified:** src/components/BoardDragLayer.tsx
- **Verification:** Overlay renders face-down correctly; pile-to-opponent pass dispatches PASS_CARD
- **Committed in:** 06aba43

**4. [Rule 1 - Bug] OpponentHand affordance only triggered for fromZone=hand, too restrictive**
- **Found during:** Task 2 (manual verification — drag from pile over opponent zone)
- **Issue:** Initial affordance implementation only showed dashed border for fromZone === 'hand'; pile drags did not trigger it despite pile-to-opponent pass being valid
- **Fix:** Changed dragIsFromHand to trigger for any active drag (active !== null) rather than checking fromZone
- **Files modified:** src/components/OpponentHand.tsx
- **Verification:** OpponentHand shows affordance during any active drag
- **Committed in:** 99a71b7
