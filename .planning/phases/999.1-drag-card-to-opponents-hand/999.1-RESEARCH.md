# Phase 999.1: Drag Card to Opponent's Hand - Research

**Researched:** 2026-04-09
**Domain:** dnd-kit droppable zones, PartyKit server authorization, cross-player hand semantics
**Confidence:** HIGH

## Summary

Phase 999.1 adds a drag-and-drop path for moving a card directly into an opponent's visible hand display. The server already has a `PASS_CARD` action that does the same thing programmatically (from CARD-04, Phase 4). This phase is primarily about wiring a drag gesture to that existing server action — not adding new server logic.

The `OpponentHand` component already has a `useDroppable` drop zone registered with `toZone: 'opponent-hand'` and `toId: playerId`. `BoardDragLayer` already detects `isPassCard` drops and dispatches `PASS_CARD`. The server already guards against unauthorized cross-player hand writes; the existing `UNAUTHORIZED_MOVE` guard in `MOVE_CARD` is what currently blocks the `toZone === 'hand' && toId !== senderToken` case — but `PASS_CARD` bypasses this by being its own action type with explicit sender ownership semantics.

The key open question is UX: what draggable source initiates this interaction? Cards in the player's own hand are currently rendered as `SortableHandCard` (using `useSortable`), and dragging one onto an opponent drop zone works in principle — `useSortable` extends `useDraggable`, so the card already participates in the dnd-kit drag context. The collision detection (`closestCenter`) may need tuning to prefer the opponent zone over the hand's own sortable items when dragging near the top of the screen.

**Primary recommendation:** Re-use `PASS_CARD` server action and the existing `opponent-hand` droppable — the end-to-end pipe is already in place. The plan work is (1) verifying the drag path works end-to-end with real UI testing, (2) adding a visual drag affordance on `OpponentHand` while a card is being dragged, and (3) writing a server-side unit test covering the drag-initiated-pass path.

## Architecture Patterns

### Existing Pattern: Drag → BoardDragLayer → sendAction

The full drag-to-opponent-hand pathway exists today:

```
SortableHandCard (useSortable, data: { card, fromZone:'hand', fromId:playerId })
    ↓ drag over
OpponentHand (useDroppable, data: { toZone:'opponent-hand', toId:opponentId })
    ↓ DragEnd event
BoardDragLayer.handleDragEnd → detects isPassCard → sendAction({ type:'PASS_CARD', ... })
    ↓ WebSocket
party/index.ts PASS_CARD handler → moves card between hands → broadcastState()
    ↓ STATE_UPDATE
All clients see updated opponentHandCounts and myHand
```

Each link in this chain is already implemented. The question is whether the seam between `useSortable` items and the opponent-hand droppable has a collision detection or visual feedback gap.

### Pattern: toZone Discrimination in BoardDragLayer

`handleDragEnd` already branches on `isPassCard`:

```typescript
// Source: src/components/BoardDragLayer.tsx
const isPassCard = dragDataRef.current?.fromZone === 'hand' && overData?.toZone === 'opponent-hand';
```

This is the existing branch. No logic change needed here — the correct action is already dispatched.

### Pattern: OpponentHand Already Has Drop Zone

```typescript
// Source: src/components/OpponentHand.tsx
const { setNodeRef, isOver } = useDroppable({
  id: `opponent-hand-${playerId}`,
  data: { toZone: 'opponent-hand' as const, toId: playerId },
});
```

The drop target exists. `isOver` is tracked but the visual feedback (border highlight) is already implemented.

### Anti-Patterns to Avoid

- **Modifying MOVE_CARD to allow cross-player toZone=hand:** The existing `UNAUTHORIZED_MOVE` guard exists for good reason — it prevents a malicious client from writing to another player's hand via MOVE_CARD. Do not relax this. Use PASS_CARD, which is a semantically correct "give a card" action.
- **Adding a new server action for drag-to-opponent:** PASS_CARD already does this. A new action would be duplicate logic and diverge the undo behavior.
- **Using `pointer-events: none` on opponent cards during drag:** This would break the droppable zone. The drop zone ref is on the container div, not the card backs.

## Common Pitfalls

### Pitfall 1: SortableContext intercepts the drag before it reaches OpponentHand
**What goes wrong:** The `SortableContext` in `HandZone` uses `closestCenter` which tries to find the nearest sortable item. When dragging to the top of the screen toward `OpponentHand`, the nearest sortable item in the hand may win the collision, causing a reorder instead of a pass.
**Why it happens:** `SortableContext` items are all registered droppables. `closestCenter` picks among ALL registered droppables, not just those in the same context.
**How to avoid:** Test this explicitly. If collision fires on the wrong target, consider `closestCorners` or a custom `collisionDetection` that gives explicit priority to `opponent-hand-*` droppables. Alternatively, the spatial separation (opponent hand is at top, player hand is at bottom) may be sufficient that `closestCenter` naturally resolves correctly.
**Warning signs:** During testing, dropping onto `OpponentHand` triggers a `REORDER_HAND` instead of `PASS_CARD`.

### Pitfall 2: isHandToHand check in BoardDragLayer swallows pass drops
**What goes wrong:** The `isHandToHand` check (`fromZone === 'hand' && toZone === 'hand'`) must not match `opponent-hand` drops. Currently it correctly checks `toZone === 'hand'` (not `opponent-hand`), so this is fine — but any refactor of these conditions must preserve the ordering.
**Why it happens:** The condition evaluation order matters: `isPassCard` is checked before `isSuccess` before `isHandToHand`.
**How to avoid:** Read `BoardDragLayer.handleDragEnd` before touching it. The ordering is already correct.

### Pitfall 3: Visual affordance missing during drag
**What goes wrong:** The `OpponentHand` zone highlights on hover (`isOver`) but players may not know it's a valid drop target during a drag. No hover state appears until the cursor enters the zone.
**Why it happens:** There's no "drag is active" state propagated to `OpponentHand`.
**How to avoid:** Consider adding a subtle highlight to all opponent hands when a drag from the player's own hand is in progress. Use `useDndContext().active` in `OpponentHand` to detect an active drag and render a visual affordance. This is a UX enhancement, not a bug, but missing it makes the feature hard to discover.

### Pitfall 4: Undo after drag-to-opponent restores sender's hand but not recipient's
**What goes wrong:** `PASS_CARD` calls `takeSnapshot` which deep-clones the whole GameState. `UNDO_MOVE` restores it. This works correctly — the full state snapshot captures both hands. No issue here, but it must be verified.
**Why it happens:** Not an issue, but worth testing.
**How to avoid:** Add an undo test case covering PASS_CARD (or verify existing undo test covers it).

## Open Questions

1. **Does closestCenter resolve correctly between HandZone sortables and OpponentHand droppable?**
   - What we know: All are registered droppables under the same `DndContext`. Spatial distance drives the algorithm.
   - What's unclear: Whether the vertical gap between hand (bottom) and opponent area (top header) is sufficient to prevent collision ambiguity at intermediate positions.
   - Recommendation: Manual test with two players. If collision misfires, try `closestCorners` or a custom collision that filters `opponent-hand-*` by proximity threshold.

2. **Should dragging from a pile to an opponent's hand also be supported?**
   - What we know: The backlog item says "drag a card directly into an opponent's hand" — context implies from own hand. Pile-to-opponent-hand would require a new `BoardDragLayer` branch since pile cards use `fromZone: 'pile'` and `isPassCard` currently only matches `fromZone: 'hand'`.
   - What's unclear: Whether this is in scope.
   - Recommendation: Start with hand-to-opponent-hand only (matches the "dealer-style" framing). Pile-to-opponent-hand can be a follow-on.

3. **Should the drag-to-opponent path be blocked when a player only has one card?**
   - What we know: No restriction exists today on PASS_CARD.
   - Recommendation: No change — consistent with existing behavior.
