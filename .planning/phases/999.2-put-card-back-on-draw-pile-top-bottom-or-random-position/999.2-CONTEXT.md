# Phase 999.2: Put Card Back on Draw Pile — Top, Bottom, or Random - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

When a player drops a card onto any pile, a post-drop dialog appears letting them choose where to insert it: Top, Bottom, or Random. Server handles insertion at the correct position. This applies to all pile drops (not just the draw pile).

**In scope:**
- Post-drop insertion position dialog (Top / Bottom / Random) for all pile drops
- Server-side `insertPosition` field on MOVE_CARD action
- `unshift()` for bottom, `push()` for top (current behavior), `splice(randomIdx, 0, card)` for random
- Dialog dismissal defaults to Top (preserves current behavior)

**Out of scope:**
- Per-pile configuration of whether position dialog shows
- Revealing random insert position to any player
- Pre-drop affordances (hover zones, modifier keys)
- Context menus

</domain>

<decisions>
## Implementation Decisions

### Position Selection UX
- **D-01:** Mechanism: post-drop dialog. Card is dropped on a pile, then a small modal/popover appears with three buttons: Top, Bottom, Random.
- **D-02:** Dialog uses existing Popover or AlertDialog component patterns (same as Deal popover and Reset confirmation dialog already in the codebase).
- **D-03:** Dismissing the dialog (Escape or click outside) defaults to Top — the current pile-push behavior. No card limbo or cancellation.

### Scope
- **D-04:** Position selection applies to ALL pile drops — any card dropped on any pile (draw, discard, custom) triggers the dialog. Consistent behavior across all pile interactions.

### Random Insertion
- **D-05:** Random = server inserts the card at a cryptographically random index within the destination pile's cards array. No player (including the sender) is told the position. Treated like sliding a card blindly into a shuffled deck.

### Default Behavior
- **D-06:** When dialog is dismissed without selecting, card goes to Top — preserving current `dest.push(card)` behavior. No breaking change for players who ignore the prompt.

### Server Change
- **D-07:** MOVE_CARD action gains an optional `insertPosition?: 'top' | 'bottom' | 'random'` field. Server default (when field is absent or undefined) is `'top'` — backward compatible with any inflight actions that don't include it.

### Claude's Discretion
- Exact dialog component choice (Popover anchored to the pile vs. centered AlertDialog)
- Button styling within the dialog
- Whether the dialog position-anchors to the drop target pile or appears in a fixed/centered location

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Popover` / `PopoverContent` / `PopoverTrigger` (src/components/ui/popover.tsx): Used by Deal button — anchors to a trigger element, appropriate for a post-drop positional choice
- `AlertDialog` suite (src/components/ui/alert-dialog.tsx): Used by Reset confirmation — full-screen modal, may be overkill for a 3-button pick but available
- `dest.push(card)` in `party/index.ts`: The single line to replace with position-aware logic

### Established Patterns
- Server-side array mutation: all card placement is `push(card)` to the destination array — `unshift()` and `splice()` for bottom/random follow the same pattern
- Action type extension: MOVE_CARD is already an object shape; adding an optional field is backward compatible
- `crypto.getRandomValues` is already used for shuffle — same approach applies for random insertion index

### Integration Points
- `BoardDragLayer.handleDragEnd` is where MOVE_CARD is dispatched — dialog state would intercept the `isSuccess` path before sending the action, waiting for user position pick
- After position is selected, the existing `sendAction({ type: 'MOVE_CARD', ... })` call fires with the added `insertPosition` field

</code_context>
