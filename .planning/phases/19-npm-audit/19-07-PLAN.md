<objective>
Close the two remaining UAT gaps from `19-HUMAN-UAT.md` (Gap 5: hamburger menu scrolls with content; Gap 7: desktop vertical overflow). Both originate from the same architectural problem: the root div owns vertical overflow AND has `overflow-x-hidden`, which (a) breaks `position:sticky` on the header per CSS spec §9.4.3 (any non-`visible`/`clip` overflow on an ancestor disables sticky), and (b) lets the unbounded body content push the HandZone off the bottom of the desktop viewport.

The fix is a single-file layout restructure of `src/components/BoardView.tsx`:

1. **Root div:** Drop `overflow-y-auto` (and `sm:overflow-hidden`) — keep ONLY `overflow-x-hidden`. The root is `h-screen flex flex-col`; with no overflow rule on the y axis, children must fit (which the new inner scroll container ensures). This makes the root no longer a scroll container and removes the sticky-blocking ancestor.
2. **Header bar:** Remove `sticky top-0 z-20` (Plan 06's failed attempt). The header is the FIRST flex child of the root and naturally has its own intrinsic height (no `flex-1` claim, no `flex-shrink` shrinking it below content); it stays anchored at the top because it sits above a `flex-1 min-h-0` sibling that absorbs all remaining space. ConnectionBanner stays where it is — it's the very first child and remains above the header in both DOM and visual order.
3. **NEW inner scroll container:** Insert a wrapping `<div className="flex-1 min-h-0 overflow-x-hidden overflow-y-auto sm:overflow-hidden flex flex-col">` around the existing center row, mySpreadZone block, and HandZone IIFE. This is the new scroll container — phone scrolls inside it; desktop is `overflow-hidden` (so `flex-1 min-h-0` enforces fitting in the remaining viewport).
4. **Center pile/communal row:** Add `min-h-0` to the existing `flex-1 flex items-center px-4 gap-4`. Without `min-h-0`, a `flex-1` item refuses to shrink below its content height — that is precisely what makes the desktop layout overflow today even after the inner container is in place.
5. **mySpreadZone wrapper:** Add `flex-shrink-0` to the existing `bg-card px-4 py-2`. This prevents the personal spread zone from being vertically squeezed when the inner scroll container is `overflow-hidden` on desktop, and ensures the HandZone (also flex-shrink-0 by its fixed-height nature) stays anchored at the bottom.

All other behavior is preserved exactly: the `opponentCount` derivation, the adaptive opponent column ternary, the opponents-row wrapper, the ControlsBar render position, the PileZone/SpreadZone/HandZone children themselves, and all imports.

Purpose: LAYOUT-04 SC2 ("all zones remain visible and operable at 375px") still requires the hamburger to be reachable while scrolling — Plan 06's `sticky top-0` was the right intent but was silently disabled by the root's `overflow-x-hidden`. The structural fix is a smaller architectural correction than trying to patch sticky with `overflow-clip` (which has weaker browser support and odd interaction with horizontal scroll).

Output: One modified file (`src/components/BoardView.tsx`) with five exact edits in the JSX tree (root div, header div, NEW inner wrapping div opening, center row className, mySpreadZone wrapper className) plus one closing `</div>` for the new inner wrapper. Zero new imports, zero prop changes, zero changes to any other file.

  scroll container in document order. Scrolling happens INSIDE inner-scroll, which is below
  the header. The header does not move because it is not in the scroll container.
- No `sticky` is needed; flex-layout anchoring is more robust than `position: sticky` (which
  was failing per CSS spec §9.4.3 due to the previous root's overflow-x-hidden).

Why this works on desktop (Gap 7 fix):
- Root is `h-screen` so it equals viewport height. Inner-scroll is `flex-1 min-h-0

- The root no longer needs an explicit overflow-y rule because (a) we explicitly removed the

  children whose total height fits the viewport (enforced by inner-scroll's flex-1 min-h-0
  on desktop) means there is no overflow to clip. Keeping `sm:overflow-hidden` would be
  defensive but adds noise; the cleaner expression is "root only governs horizontal".
