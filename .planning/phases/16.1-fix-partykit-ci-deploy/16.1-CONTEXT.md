# Phase 16.1: fix-partykit-ci-deploy - Context

**Gathered:** 2026-05-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a `partykit deploy` step to the existing GitHub Actions `deploy.yml` workflow so that every push to main deploys both the Vite frontend (GitHub Pages) and the PartyKit server in a single job, keeping them in sync.

</domain>

<decisions>
## Implementation Decisions

### Workflow Structure
- **D-01:** Add `npm run deploy` (partykit deploy) as a step inside the existing `build-and-deploy` job — do NOT create a separate job.

### Deploy Ordering
- **D-02:** `npm run deploy` runs **after** `npm run build` but **before** `actions/configure-pages`, `actions/upload-pages-artifact`, and `actions/deploy-pages`. Server goes live first; when the new Pages client arrives, the new server is already running.

### Failure Coupling
- **D-03:** No `continue-on-error` on the partykit deploy step. Default same-job behavior: if `npm run deploy` fails, the job halts and the Pages deploy never runs. Both deployments succeed together or neither does.

### Token Setup (manual prerequisite)
- **D-04:** `PARTYKIT_TOKEN` must be stored as a **repo-level** GitHub Actions secret (not environment-scoped). The executor should document this as a manual setup step in the PR description or verification notes — it cannot be automated.

</decisions>
