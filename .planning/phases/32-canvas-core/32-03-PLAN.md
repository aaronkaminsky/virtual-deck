  <what-built>
    Phase 32 (Plans 01 + 02 combined) shipped:
    - Server: `canvasCards: CanvasCard[]` on GameState, `PLACE_ON_CANVAS` action, `MOVE_CARD fromZone:'canvas'`, `RESET_TABLE` canvas sweep, `viewFor()` broadcasts canvas, `onStart` migration
    - Client: `CanvasZone` + `CanvasDraggableCard` components, customCollision canvas fallback, handleDragEnd canvas branch dispatching PLACE_ON_CANVAS, PendingMove canvas-source extension, DragOverlay at 0.5 opacity + scale 1.05

    Net effect: Players can free-drag cards to any position on the canvas. Server stores (x, y, z) per canvas card. Other players see the canvas update in real time. Cancel and missed drops leave the card at its prior position with zero risk of card loss.
  </what-built>
  <how-to-verify>
    **SETUP**

    1. Start the local stack in two terminals:
       - Terminal A: `npm run dev` (PartyKit server on port 1999)
       - Terminal B: `npm run dev:client` (Vite on port 5173)

    2. Open TWO separate browser contexts (NOT two tabs in the same profile — use one regular window and one incognito window, or two profiles). This ensures independent localStorage so the two windows join as DIFFERENT players. Open `http://localhost:5173` in each.

    3. In window P1: enter display name "Alice", join a room (note the room URL — copy it). In window P2: paste the room URL, enter display name "Bob", join. Both should land on the board with their own hand visible and the opponent's hand visible at the top.

    4. P1 deals 5 cards: open the controls popover (hamburger icon), set deal count to 5, click Deal. Both windows now have hand cards.

    **CHECKLIST — VERIFY EACH ITEM IN ORDER**

    For each item, write PASS or FAIL with a one-line note in the SUMMARY at the end. Stop and report immediately if any FAIL.

    **Desktop viewport (1280x720 — set browser window size accordingly)**

    1. **CANVAS-01 hand → canvas**: In P1, drag any hand card to the canvas (the large area to the right of the draw/discard sidebar). Release the pointer.
       - Expected: card appears on the canvas at approximately where the pointer was released; the canvas drop target showed a faint orange ring (`ring-primary/30`) while hovering.
       - Verify in P2: the same card appears at the same position on P2's canvas (real-time sync).

    2. **CANVAS-01 pile → canvas**: In P1, drag the top card of the discard pile (or draw pile if discard is empty — discard requires placing one first) to the canvas. Release in a different spot than step 1.
       - Expected: card moves from pile to canvas; pile count decrements; canvas card appears at the new position.
       - Verify in P2: the change syncs.

    3. **CANVAS-03 z-ordering on initial placement**: After steps 1 and 2, drag a third card onto the canvas SO IT OVERLAPS the cards from steps 1 and 2. Release.
       - Expected: the newest card visually renders ON TOP of the overlapping cards. (z = max + 1 — Phase 33 will handle pointer events and shadows; for now we only verify visual stack order.)

    4. **CANVAS-02 canvas → canvas reposition**: In P1, drag one of the placed canvas cards to a new position on the canvas. Release.
       - Expected: card moves to the new position; no duplicate, no removed card.
       - Verify in P2: the card moved to the same position in P2's view.

    5. **CANVAS-04 z-ordering on canvas → canvas**: Place card A on the canvas. Then place card B on top of A. Then drag card A to overlap B. After release:
       - Expected: card A now renders on top of B (because A's z was bumped to max+1 on the canvas → canvas move).

    6. **CANVAS-02 / NOLOSS-01 Escape during drag**: In P1, start dragging a canvas card. While the card is being dragged (overlay visible), press Escape.
       - Expected: the card returns to its prior position. No PLACE_ON_CANVAS dispatched. No card disappears. P2's view is unchanged.

    7. **NOLOSS-01 drop outside any valid zone**: In P1, start dragging a canvas card. Drag the pointer OUTSIDE the canvas area AND outside any hand/pile/spread zone (e.g. above the page, into the connection banner). Release.
       - Expected: card returns to its prior canvas position. No PLACE_ON_CANVAS dispatched.

    8. **NOLOSS-01 drop outside for hand card**: In P1, start dragging a hand card. Release it in a dead zone (e.g. between the canvas and the spread zone, on no droppable). Expected: card returns to hand; no action dispatched (preserves prior hand-card behavior).

    9. **Canvas → pile dialog (D-11)**: In P1, drag a canvas card onto the NON-EMPTY discard pile (place a card there first if needed). Release on the pile.
       - Expected: the existing Top/Bottom/Random insert dialog appears (same UX as hand → pile). Click "Top".
       - Expected after confirmation: card moves from canvas to top of discard pile; canvas no longer shows that card; discard count increases.
       - Verify in P2: change syncs.

    10. **Canvas → pile dialog — cancel via Escape**: Repeat step 9 but press Escape when the dialog appears.
        - Expected: dialog dismisses; the canvas card RETURNS to its original canvas position (since MOVE_CARD was never dispatched, server state unchanged). No card lost.

    11. **DragOverlay visual contract (D-13)**: Watch carefully during any drag. The DragOverlay (the ghost following the pointer) should render at approximately 50% opacity and very slightly scaled up. The source card on the canvas/hand/pile should be HIDDEN (opacity: 0) during the drag, not visible underneath.

    12. **RESET_TABLE canvas sweep (success criterion 4 + MIGRATE-03)**: With several cards on the canvas, click Reset Table in the controls popover. Confirm any dialog.
        - Expected: canvas clears (all cards removed); draw pile is restored with all 52 cards (face down); hands and other piles empty.
        - Verify in P2: same state.

    13. **Undo PLACE_ON_CANVAS**: After step 1, click the Undo button.
        - Expected: the card returns from canvas to its original hand position; canvas removes the card.

    **Mobile viewport (375x667 — use browser devtools responsive mode or resize to 375 wide)**

    14. **CANVAS-01 hand → canvas at mobile width**: Repeat step 1 at 375px width.
        - Expected: drag works; card lands on the (smaller) canvas; clamp prevents the card going off-screen even if the pointer is released near the edge.

    15. **Position clamping at mobile width (D-15)**: Drag a card and release with the pointer near the FAR RIGHT edge of the canvas.
        - Expected: card's left position is clamped so the card's right edge does not exceed the canvas's right edge. Card dimensions at mobile are 42x59px.

    16. **No layout collapse**: After step 14, verify that the personal spread zone and hand zone are still visible below the canvas (canvas height did not push them off-screen). Note: full mobile edge-pan + bounded canvas height is Phase 35, not Phase 32 — only verify nothing is broken/inaccessible at this viewport.
