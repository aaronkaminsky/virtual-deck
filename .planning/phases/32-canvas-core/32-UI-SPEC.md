## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding inside cards |
| sm | 8px | Compact element spacing, pile zone internal padding |
| md | 16px | Default element spacing, canvas padding inset |
| lg | 24px | Section padding, sidebar vertical gap |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions: Canvas card position values are arbitrary pixel coordinates (x, y), not constrained to the spacing scale. Card clamp boundaries use card dimensions: mobile 42px wide × 59px tall; desktop 63px wide × 88px tall (source: CardFace.tsx).

## Interaction Contract

### Drag Activation

- `PointerSensor` with `activationConstraint: { distance: 8 }` (source: CONTEXT.md D-16, spike CONVENTIONS.md)
- Prevents drag activation on tap/click; allows flip-on-click to work correctly
- Touch supported: `touchAction: 'none'` on draggable card element

### Canvas Drop Target

- Canvas div registered as `useDroppable({ id: 'canvas' })` (source: CONTEXT.md D-08)
- Drop target has no visual boundary at rest — canvas is flush with board background
- On `isOver: true`: apply `ring-1 ring-primary/30` outline to canvas border (1px, accent at 30% opacity)
- `event.over?.id === 'canvas'` in `handleDragEnd` dispatches `PLACE_ON_CANVAS`

### Cancel / No-Op Drop Behavior

- `onDragCancel`: no action sent, no state mutation; card stays at original position (source: CONTEXT.md D-14)
- `event.over === null` (missed drop for canvas cards): no action sent; card stays at canvas position. NOLOSS-01 satisfied by this no-op (source: CONTEXT.md D-09)
- `event.over === null` for hand/pile cards: existing behavior unchanged

### Position Clamping

- On drop: `newX = clamp(x + delta.x, 0, canvasWidth - CARD_W)`, `newY = clamp(y + delta.y, 0, canvasHeight - CARD_H)` (source: CONTEXT.md D-15)
- `CARD_W` / `CARD_H`: 63px / 88px at `sm:` breakpoint and above; 42px / 59px below `sm:` (source: CardFace.tsx)
- Canvas dimensions measured via element ref on `DndContext` drag end

### Z-Ordering

- Drop on canvas: server sets `z = max(existing canvasCards z) + 1` (source: CONTEXT.md D-07, CANVAS-04)
- CSS `zIndex` on `CanvasDraggableCard` set to `card.z` from `ClientGameState.canvasCards`

### Insert Dialog (canvas → pile)

- When canvas card dropped onto draw or discard pile: show existing top/bottom/random insert dialog
- Same UX as hand→pile drops; uses `MOVE_CARD` with `fromZone: 'canvas'` (source: CONTEXT.md D-11)
- No new dialog design needed

### Click-vs-Drag Guard

- Mirror `DraggableCard.tsx` pattern: `didDragRef` + 300ms timeout (source: spike CONVENTIONS.md)
- Phase 32 canvas cards are always face-up (no flip action for canvas cards in this phase; per CONTEXT.md out-of-scope list)

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | No new CTA in this phase — all actions initiated by drag |
| Empty state heading | (canvas empty) — no heading displayed; canvas shows only the felt background |
| Empty state body | No copy — empty canvas is a valid game state, not an error |
| Error state | "Could not place card — try again" (shown if server rejects PLACE_ON_CANVAS; inline toast or silent revert) |
| Destructive confirmation | None in this phase |

Note: Canvas has no "no cards" prompt. An empty canvas communicates "the table is clear" without text. This is intentional per the physical card table metaphor.

## Accessibility Contract

- Canvas droppable: `aria-label="Play area"` on the canvas div
- `CanvasDraggableCard`: `aria-roledescription="Draggable card"`, `aria-label="{rank} of {suit}"` (derived from card data)
- dnd-kit provides keyboard drag-and-drop natively via `attributes` spread; `aria-pressed` override placed after `{...attributes}` spread (source: CLAUDE.md dnd-kit convention)
- Canvas cards render in z-index order (low to high in DOM); screen readers announce in DOM order, which matches visual z-stack
