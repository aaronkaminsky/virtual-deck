## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client UI→server | Grid cell droppable data (toRow, toCol, toId) originates from client-side useDroppable data payloads; the server validates these values independently per Plan 01 T-24-01/T-24-02 |
| drag event data | activeData.card, fromZone, fromId originate from useDraggable data set at drag start; no server trust issues, client-only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-05 | Tampering | BoardDragLayer — `isIntraSpreadReorder` suppression for intra-grid drags | mitigate | Grid cell droppable `data.toId` is set to the pile ID (`'play'`), not the cell ID — this makes `fromIdAtEnd === overData?.toId` true for intra-grid drops, suppressing MOVE_CARD in BoardDragLayer; GridZone useDndMonitor fires MOVE_GRID_CARD instead. Verify this in GridCell data payload. |
| T-24-06 | Tampering | customCollision — grid-cell ID prefix collision with pile-* bucket | mitigate | Grid cell IDs use prefix `grid-cell-` which does not start with `pile-`; confirmed by literal string. If a grid-cell ID accidentally started with `pile-`, it would enter the pile collision bucket and trigger isIntraPileDrag, racing with GridZone's useDndMonitor. |
| T-24-07 | Information Disclosure | GridCell droppable data payload | accept | Cell coordinates (row, col) are visible to any client reading drag event data; this is not sensitive — grid layout is public. |
| T-24-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this phase; no registry interaction needed. |
