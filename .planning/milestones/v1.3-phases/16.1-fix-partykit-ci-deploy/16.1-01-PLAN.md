<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub repo → GitHub Actions runner | Repo-level secret `PARTYKIT_TOKEN` is injected as an env var into the runner container. Untrusted code in any PR workflow run could exfiltrate the token if exposed at the wrong scope. |
| GitHub Actions runner → PartyKit Cloud | Token authenticates `partykit deploy` against PartyKit's API. A leaked token grants full deploy access to the `virtual-deck` PartyKit project. |
| `.github/workflows/deploy.yml` → `package.json` deploy script | Workflow trusts that `npm run deploy` resolves to `partykit deploy` and not arbitrary code. A compromised `package.json` (supply chain) could exfiltrate the token via the same env var. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16.1-01 | Information Disclosure | `PARTYKIT_TOKEN` secret in workflow logs | mitigate | GitHub Actions automatically masks secret values in logs. The token is bound at the step level via `env: PARTYKIT_TOKEN: ${{ secrets.PARTYKIT_TOKEN }}` so it is only present in the `npm run deploy` step's environment, not all steps. Verified by acceptance criterion: `grep -F 'PARTYKIT_TOKEN: ${{ secrets.PARTYKIT_TOKEN }}'` matches at the step level (not job-level `env:`). |
| T-16.1-02 | Elevation of Privilege | Pull-request workflows on forks accessing `PARTYKIT_TOKEN` | mitigate | Workflow trigger is `on: push: branches: [main]` and `workflow_dispatch` only — NOT `pull_request` or `pull_request_target`. Forked PRs cannot trigger this workflow, so they cannot access the secret. Verified by inspection: deploy.yml's `on:` block contains only `push.branches: [main]` and `workflow_dispatch`. |
| T-16.1-03 | Tampering | Compromised dependency in `package.json` deploy script chain | accept | Risk inherent to any `npm ci` + `npm run` workflow. Mitigated at ecosystem level by lockfile (`package-lock.json`). Out of scope for this phase. Severity: medium; likelihood: low for a hobby project with pinned `partykit` 0.0.115. |
| T-16.1-04 | Repudiation | Unaudited deploys (who triggered which version?) | accept | GitHub Actions logs the commit SHA, actor, and timestamp for every workflow run. Repo audit log preserves this. No additional logging needed for this phase. |
| T-16.1-05 | Denial of Service | `partykit deploy` failing repeatedly halts Pages deploys (per D-03) | accept | This is the INTENDED behavior — D-03 explicitly couples failure so client and server cannot drift. Trade-off: a flaky partykit auth issue blocks frontend updates until resolved. Severity: low; recovery path is clear (re-run workflow, or rotate token). |
| T-16.1-06 | Spoofing | Attacker with GitHub repo write access pushes malicious `party/index.ts` and the workflow auto-deploys it | accept | Inherent to any auto-deploy CI. Mitigated by GitHub branch protections (configurable separately by user) and limiting repo write access. Out of scope for this phase. |
| T-16.1-07 | Information Disclosure | `PARTYKIT_TOKEN` set at job level instead of step level (would expose token to all steps including third-party actions) | mitigate | Token MUST be set on the `npm run deploy` step's `env:` block ONLY, not at the workflow-level or job-level `env:`. Acceptance criterion enforces this: `grep` checks the token reference appears alongside `npm run deploy`, and the workflow-level `env:` block contains only `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24`. |

## Residual Risks

- **Token rotation**: Manual process. If the user's PartyKit account token is compromised, they must regenerate it locally and update the GitHub secret. No automation.
- **Partial deploy window**: Per D-02, partykit deploy completes BEFORE Pages deploy starts. There is a small window (seconds) where the new server is live but the old client is still being served. Acceptable per D-02 rationale ("server goes live first; when the new Pages client arrives, the new server is already running"). Reverse drift (old server, new client) is the failure mode this phase is designed to prevent — and it cannot occur given D-03 + D-02.
</threat_model>
