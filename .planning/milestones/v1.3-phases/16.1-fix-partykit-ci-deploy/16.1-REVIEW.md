## Summary

Single workflow file reviewed: `.github/workflows/deploy.yml`. The file deploys both a PartyKit server and a GitHub Pages static frontend in a single job. Two critical security issues were found — one involving how the PartyKit token is written to disk and one involving a missing `--no-install` flag that may silently mutate the deployment environment. Two warnings address fragility in the job ordering and a missing failure gate. One info item covers a loose Node.js version pin.

## Critical Issues

### CR-01: PARTYKIT_TOKEN written as raw string — likely produces malformed config.json

**File:** `.github/workflows/deploy.yml:37-38`
**Issue:** The step writes the secret directly into `~/.partykit/config.json` with `echo "$PARTYKIT_TOKEN" > ~/.partykit/config.json`. The PartyKit CLI expects `config.json` to be a JSON object (e.g., `{"login":"...","access_token":"..."}`) not a bare token string. Writing a raw token value as the file contents will produce a file that is not valid JSON. If the CLI attempts to parse this file it will either crash with a JSON parse error or silently fall back to an unauthenticated deploy, making the step appear to succeed while actually deploying nothing or deploying anonymously.

The correct approach depends on the exact schema the PartyKit CLI uses for its config file. The safest fix is to pass the token via the environment variable that the CLI natively supports (`PARTYKIT_LOGIN` / `PARTYKIT_TOKEN` or `CLOUDFLARE_API_TOKEN` depending on PartyKit version) rather than constructing a config file by hand:

**Fix:**
```yaml
- run: npm run deploy
  env:
    PARTYKIT_TOKEN: ${{ secrets.PARTYKIT_TOKEN }}
```

If the CLI does require a config file, the value must be wrapped in a JSON structure:
```yaml
- run: |
    mkdir -p ~/.partykit
    echo '{"access_token":"'"$PARTYKIT_TOKEN"'"}' > ~/.partykit/config.json
    npm run deploy
  env:
    PARTYKIT_TOKEN: ${{ secrets.PARTYKIT_TOKEN }}
```
Verify the exact key name against the PartyKit CLI source or documentation before using either form.

### CR-02: Secret value echoed via shell — token exposed in command expansion

**File:** `.github/workflows/deploy.yml:38`
**Issue:** `echo "$PARTYKIT_TOKEN" > ~/.partykit/config.json` passes the secret through a shell `echo` command. GitHub Actions masks secret values in log output, but the masking is best-effort: it only masks the literal string that matches the secret value. If the token contains characters that are percent-encoded, whitespace-split, or otherwise transformed during shell expansion, the mask may not fire. Additionally, echoing secrets through shell commands rather than writing them via a dedicated secrets mechanism is an auditable security anti-pattern — it means the token appears in the process argument list on the runner, which is visible to any process that can read `/proc/<pid>/cmdline` on a shared Linux runner.

The fix is to avoid constructing the config file via `echo` at all. Use `PARTYKIT_TOKEN` as a native environment variable that the CLI reads directly (see CR-01 fix), or write the file using a multi-line `run` step that never places the secret in a positional argument:

**Fix:**
```yaml
- run: |
    mkdir -p ~/.partykit
    printf '%s' "$PARTYKIT_TOKEN" > ~/.partykit/config.json
  env:
    PARTYKIT_TOKEN: ${{ secrets.PARTYKIT_TOKEN }}
```
`printf '%s'` avoids the trailing newline that `echo` appends, and keeps the secret in the environment rather than a positional argument. But the preferred fix remains CR-01: pass the token as an env var the CLI already knows how to consume, and skip manual file construction entirely.

## Warnings

### WR-01: GitHub Pages deploy proceeds even if PartyKit deploy fails

**File:** `.github/workflows/deploy.yml:36-46`
**Issue:** All steps run sequentially in a single job. If `npm run deploy` (PartyKit) exits with a non-zero code, GitHub Actions will halt the job and the GitHub Pages artifact will not be uploaded. However, the inverse is not protected: if the PartyKit deploy step were to silently succeed while the server is actually broken (e.g., due to the malformed config.json in CR-01), the Pages deploy continues and ships a frontend that points at a non-functional server. More importantly, there is no explicit ordering guarantee between "frontend build uses `VITE_PARTYKIT_HOST`" and "PartyKit server is actually live at that host." A two-job structure with `needs:` would make this dependency explicit and allow the Pages deploy to be skipped if the server deploy fails.

**Fix:** Split into two jobs:
```yaml
jobs:
  deploy-partykit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: lts/*
          cache: npm
      - run: npm ci
      - run: npm run deploy
        env:
          PARTYKIT_TOKEN: ${{ secrets.PARTYKIT_TOKEN }}

  deploy-pages:
    needs: deploy-partykit
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    permissions:
      contents: read
      pages: write
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: lts/*
          cache: npm
      - run: npm ci
      - run: npm run build
        env:
          VITE_PARTYKIT_HOST: virtual-deck.aaronkaminsky.partykit.dev
      - uses: actions/configure-pages@v4
      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist
      - uses: actions/deploy-pages@v4
        id: deployment
```

### WR-02: `npm run deploy` (partykit deploy) runs without `--no-install` — may modify lockfile in CI

**File:** `.github/workflows/deploy.yml:39`
**Issue:** `partykit deploy` by default runs its own dependency installation step in the worker build context. In CI, `npm ci` has already installed a clean, lockfile-verified environment, but `partykit deploy` may invoke `npm install` internally during the build, bypassing the lockfile and pulling undeclared transitive dependencies. This silently breaks the "reproducible build" guarantee that `npm ci` provides. Additionally, any `npm install` invocation that updates `package-lock.json` inside the runner will cause the cache to diverge from the committed lockfile without any warning.

**Fix:** Pass the `--no-install` flag if the PartyKit CLI supports it (check `partykit deploy --help`), or confirm via PartyKit documentation that the deploy command uses the already-installed `node_modules` rather than re-installing.

## Info

### IN-01: `node-version: lts/*` is a floating version pin

**File:** `.github/workflows/deploy.yml:30`
**Issue:** `lts/*` resolves to whatever the current LTS is at runtime. When a new Node.js LTS is promoted, the workflow will silently start using a different major version on the next run. For a small project this is usually fine, but it means a Node.js major version upgrade can cause an unexpected build failure with no code change. A pinned LTS version (e.g., `node-version: 22`) provides reproducibility at the cost of a one-line update when you intentionally want to upgrade.

**Fix:**
```yaml
node-version: '22'
```
This is a low-priority improvement; `lts/*` is widely accepted in hobby projects.
