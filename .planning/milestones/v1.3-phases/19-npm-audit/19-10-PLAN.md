<objective>
Close Gap 10: when the communal play area has many cards and the board layout recalculates, the opponents row (which has overflow-x-auto) involuntarily scrolls to the right, hiding one opponent column with no visible way to scroll back.

Root cause: the opponents row uses `overflow-x-auto`, which allows horizontal scroll. When any layout pressure causes horizontal overflow in that row, the browser may auto-scroll it. The user has no scroll affordance to navigate back.

Fix: two changes to BoardView.tsx:

1. **Opponents row**: change `overflow-x-auto` → `overflow-hidden`. The opponents row should never scroll — each column must fit within the available width. If content overflows inside a column, the column's own `overflow-x-hidden` clips it.

2. **Per-opponent column className**: change the multi-opponent branch from `max-w-[200px]` to `flex-1 min-w-0`. With `flex-1 min-w-0`, each of N opponent columns gets an equal share of the row width (e.g., 2 opponents each get ~50%). `min-w-0` is required to allow flex items to shrink below their content's intrinsic width. The single-opponent branch (`flex-1 max-w-none`) is unchanged.

The resulting ternary for the per-opponent column wrapper:
```
opponentCount === 1 ? 'flex-1 max-w-none' : 'flex-1 min-w-0'
```

At desktop `sm:max-w-none` on the column wrapper ensures no cap applies at large sizes (unchanged).
</objective>
