<objective>
Close GAP-01 from Plan 03 human verification: dragging a card from the discard pile down
even a tiny bit causes the player hand drop zone to light up and accept the drop before
the pointer actually enters the visible hand strip rect.

This plan investigates the current collision detection wiring (already `pointerWithin` for
zone containers per BoardDragLayer.tsx line 22, but the gap persists in human testing),
identifies the actual cause of early activation, and applies the minimal correct fix.

Two suspect mechanisms:
  (1) The hand droppable's DOM rect is larger than its visible h-[100px] / sm:h-[128px]
      strip — e.g. a wrapper or MeasuringStrategy.Always caching a stale taller rect.
  (2) `isOver` for hand-zone styling in HandZone.tsx (lines 137-141) is computed from
      `useDndContext().over` and treats ANY hand-card sortable id as the over target.
      For a CROSS-ZONE drag (fromZone !== 'hand'), customCollision does NOT query card
      containers unless inside a zone, so a hand card should not be `over` until inside the
      zone — BUT if the hand droppable's measured rect is inflated, the zone "wins" first
      anyway. Verify which path triggers in human testing before patching.

Purpose: Make the hand drop zone activate iff the pointer is inside the visible hand strip
rect, for ALL drag sources (discard, draw, personal spread, opponent spread, hand reorder).

Output: customCollision and/or HandZone updated so a discard→hand drag with the pointer
only a few pixels below the discard pile does NOT register the hand zone as `over` and does
NOT visually highlight the hand. Existing automated tests stay green.
</objective>

src/components/BoardDragLayer.tsx — customCollision (lines 11-55):

```
const customCollision: CollisionDetection = (args) => {
  const zoneContainers = args.droppableContainers.filter(
    (c) => String(c.id) === 'hand' || String(c.id).startsWith('opponent-hand-')
  );
  const pileContainers = args.droppableContainers.filter(
    (c) => String(c.id).startsWith('pile-')
  );
  const cardContainers = args.droppableContainers.filter(
    (c) => String(c.id) !== 'hand' && !String(c.id).startsWith('opponent-hand-') && !String(c.id).startsWith('pile-')
  );

  const zoneCollisions = pointerWithin({ ...args, droppableContainers: zoneContainers });

  if (zoneCollisions.length > 0) {
    if (String(zoneCollisions[0].id).startsWith('opponent-hand-')) return zoneCollisions;
    const activeData = args.active.data.current as { fromZone?: string } | undefined;
    if (activeData?.fromZone === 'hand') {
      const cardCollisions = closestCenter({ ...args, droppableContainers: cardContainers });
      return cardCollisions.length > 0 ? cardCollisions : zoneCollisions;
    }
    return zoneCollisions;
  }

  const pileCollisions = pointerWithin({ ...args, droppableContainers: pileContainers });
  if (pileCollisions.length > 0) {
    const activeData = args.active.data.current as { fromZone?: string; fromId?: string } | undefined;
    const isIntraPileDrag = activeData?.fromZone === 'pile' &&
      pileCollisions.some(c => String(c.id) === `pile-${activeData?.fromId}`);
    if (isIntraPileDrag) {
      const cardCollisions = closestCenter({ ...args, droppableContainers: cardContainers });
      return cardCollisions.length > 0 ? cardCollisions : pileCollisions;
    }
    return pileCollisions;
  }
  return [];
};
```

Key facts established by reading the code:
- The hand droppable `setNodeRef` is attached to the inner `h-[100px] sm:h-[128px]` div (line 244). Its DOM rect should match the visible strip.
- `MeasuringStrategy.Always` (set on the DndContext in BoardDragLayer line 352) measures every drag move, so a stale-rect cause is unlikely.
- HandZone hand cards register as droppables via `useSortable`. These card droppables are filtered into `cardContainers` (BoardDragLayer line 18-20).
- customCollision ONLY queries cardContainers when zoneCollisions returns at least one hit AND activeData.fromZone === 'hand' (line 31). A discard→hand drag has activeData.fromZone === 'pile', so this branch is not taken.
- BUT: the `useDndContext().over` value in HandZone (line 137) reads the FIRST collision returned by customCollision. If customCollision returns a hand-card id (only possible via intra-hand drag), `over.id` is a card id, and `handCardIds.has(...)` is true → hand `isOver` styling fires. For a CROSS-ZONE drag this code path cannot fire because customCollision never returns a card id when fromZone !== 'hand'.
- Therefore the visible "early activation" must mean: customCollision returns `over.id === 'hand'` while the pointer is still outside the visible hand strip rect. `pointerWithin` uses the droppable's CACHED rect (refreshed by MeasuringStrategy.Always on each drag move). Either the rect is genuinely inflated, OR dnd-kit's pointer coordinates are offset from the visible rect.

In that case the fix is in HandZone's isOver derivation.

Based on investigation outcome, apply ONE of the three fixes below in Task 1. Document
which fix path was taken and why in the SUMMARY.

Fix A — Tighten HandZone isOver to require an actual zone match (no card-id fallback):
- Change isOver derivation to ONLY trust over.id === 'hand', removing the `handCardIds.has(...)` fallback. Rationale: with customCollision's current gating, the hand zone is the canonical drop target for any cross-zone drag; hand-card ids are only `over` during intra-hand sortable reorder, which uses its own visual feedback (the dragged card placeholder). The `border-t-2 border-primary` strip is a CROSS-ZONE drop affordance and should only fire for `over.id === 'hand'`. This removes the only Documented Vector for early hand styling.

Fix B — Switch the hand droppable to a tighter rect using useDroppable's resizeObserverConfig or a custom rect computation (more invasive, lower priority).

Fix C — Add a customCollision guard that ignores the hand zone when the active drag's pointer is outside the strict h-[100px]/sm:h-[128px] visible band. (Most invasive; only if Fix A is insufficient.)

Default expectation: Fix A is the correct minimum-invasive fix. Confirm with the
investigation steps before applying. If the visible-rect bounding test (Step I-2)
shows an inflated rect, additionally apply Fix C; do not apply Fix B unless the
executor confirms it is necessary.
