## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing |
| md | 16px | Default element spacing |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Canvas PADDING constant (inner canvas overflow pad) |
| 3xl | 64px | Page-level spacing |

Exceptions:
- **EdgeArrow touch target:** 32×80px (left/right arrows) or 80×32px (top/bottom arrows). These are non-square targets sized for thumb reach, not the 8-point scale. Source: 35-CONTEXT.md D-09 / Spike003 validated values.
- **Canvas PADDING constant:** 48px (maps to `2xl` token). This is the overflow buffer added beyond the rightmost/bottommost card position when computing inner canvas bounds. Source: 35-CONTEXT.md D-02.

## Typography

Typography is unchanged from the existing application. Phase 35 introduces no new text elements beyond the arrow glyph.

| Role | Size | Weight | Line Height | Source |
|------|------|--------|-------------|--------|
| Body | 16px | 400 (regular) | 1.5 | src/globals.css (Inter default) |
| Label | 14px | 400 (regular) | 1.4 | existing UI labels |
| Heading | 20px | 600 (semibold) | 1.2 | existing section headings |
| Display | 28px | 600 (semibold) | 1.2 | existing lobby/room headings |

**Arrow glyph:** `‹` character, `font-size: 20px`, `color: white`, rotated per direction. Not a typography decision — purely a glyph reused as a visual arrow indicator. Source: 35-CONTEXT.md D-09 / Spike003.

Rotation map (validated in spike, must not deviate):
- `left: 0deg`, `right: 180deg`, `up: 90deg`, `down: 270deg`
- Do NOT use `›` as base for right/down — doubling-flip bug documented in spike README.

## Color

All values sourced from `src/globals.css` `:root` block. Phase 35 introduces one new semi-transparent surface (EdgeArrow background) that is hardcoded per spike validation — it intentionally does not use a CSS variable.

| Role | Value | Usage | Source |
|------|-------|-------|--------|
| Dominant (60%) | `hsl(160 38% 16%)` — `var(--background)` | Canvas background, page background | globals.css |
| Secondary (30%) | `hsl(160 38% 10%)` — `var(--card)` | Sidebar, card surfaces, header bar | globals.css |
| Accent (10%) | `hsl(38 92% 50%)` — `var(--primary)` | Canvas drop ring (`ring-primary/30`), focus rings, active UI elements | globals.css |
| Destructive | `hsl(0 84% 60%)` — `var(--destructive)` | Destructive actions only | globals.css |

Accent reserved for: canvas `isOver` drop ring only (existing). No new accent usage introduced in this phase.

**EdgeArrow background (hardcoded, not a CSS variable):**
- Value: `rgba(255, 255, 255, 0.15)` with `backdrop-filter: blur(4px)`
- This is a frosted-glass overlay effect on the canvas surface. Using a CSS variable would require a new token that only this one component uses. Spike003 validated this value; no change is warranted.
- `backdrop-filter: blur(4px)` — supported in all modern target browsers (Chrome, Safari, Firefox). Risk: cosmetic-only if unavailable. No fallback needed.

## Interaction Contract

This section describes the new interaction behaviors Phase 35 introduces. No equivalent section exists in the base template — added because Phase 35 is primarily an interaction phase with minimal visual surface.

### EdgeArrow

| Property | Value | Source |
|----------|-------|--------|
| Visibility trigger | Arrow appears when content overflows in that direction; hidden when no overflow | 35-CONTEXT.md D-06 |
| Overflow conditions | left: `scrollX > 0`; right: `scrollX < innerW - viewportW`; up: `scrollY > 0`; down: `scrollY < innerH - viewportH` | 35-CONTEXT.md D-06 |
| Pan speed | 8px per tick | 35-CONTEXT.md D-07 |
| Pan frequency | 16ms interval (~60fps) | 35-CONTEXT.md D-07 |
| Interaction model | `onPointerDown` starts pan; `onPointerUp` + `onPointerLeave` stop pan | 35-CONTEXT.md D-08 |
| dnd-kit isolation | `stopPropagation` on `onPointerDown` — prevents dnd-kit from seeing arrow press as drag start | 35-CONTEXT.md D-08 |
| z-index | 1000 — above all canvas cards | Spike003 |
| `touchAction` | `none` on each arrow div — prevents browser scroll capture | Spike003 |
| Viewport scope | All viewport widths (not mobile-only); arrows appear on desktop too when content overflows | 35-CONTEXT.md D-06 |

### Canvas Height Cap (mobile)

| Property | Value | Source |
|----------|-------|--------|
| Breakpoint | `< 640px` viewport width | 35-CONTEXT.md D-04 |
| CSS approach | `max-h-[Xpx] sm:max-h-none` on CanvasZone outer viewport div | 35-CONTEXT.md D-04 |
| Starting value | ~220–260px — tune in live browser at 375×667 so spread zone top is visible | 35-RESEARCH.md Open Questions |
| Clipped content access | Down-arrow pan reveals content below the cap | 35-CONTEXT.md D-05 |

### Two-Div Panning Model

| Property | Value | Source |
|----------|-------|--------|
| Outer div | `overflow: hidden`, `flex-1`, `position: relative` | 35-CONTEXT.md D-03 |
| Inner div | `position: absolute`, `width: innerW`, `height: innerH`, `transform: translate(-scrollX, -scrollY)` | 35-CONTEXT.md D-03 |
| Panning mechanism | CSS transform on inner div — NOT native scroll | 35-CONTEXT.md D-03 |
| Reason | Native `overflow: scroll` conflicts with dnd-kit `setPointerCapture` on touch; transform is validated workaround | 35-RESEARCH.md Anti-Patterns |

## Copywriting Contract

Phase 35 introduces no new user-visible copy. All existing copy is unchanged. The EdgeArrow has no label; it is a purely visual affordance (glyph + position conveys direction).

| Element | Copy | Notes |
|---------|------|-------|
| Primary CTA | n/a — no new CTAs in this phase | — |
| Empty state heading | n/a — canvas empty state unchanged | Empty canvas = no arrows, no error |
| Empty state body | n/a | — |
| Error state | n/a — no new error states | Overflow errors are silent; arrow simply disappears |
| Destructive confirmation | n/a — no destructive actions in this phase | — |

**Aria label for EdgeArrow:** Each arrow must have `aria-label` describing its direction:
- Left: `"Pan canvas left"`
- Right: `"Pan canvas right"`
- Up: `"Pan canvas up"`
- Down: `"Pan canvas down"`

This is an accessibility requirement, not a visual copywriting choice.
