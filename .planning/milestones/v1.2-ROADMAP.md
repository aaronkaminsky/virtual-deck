# Milestone v1.2: Dev Infrastructure & Game Depth

**Status:** ✅ SHIPPED 2026-04-29
**Phases:** 12–16
**Total Plans:** 14 (12: 1, 13: 3, 14: 6, 15: 3, 16: 1)

## Overview

Established developer tooling and added gameplay zones with multi-card set support for 13s-style games. This milestone introduced a full Playwright e2e infrastructure, shared test helpers proving hand-masking correctness, personal and communal spread zones visible to all players, and atomic multi-card set play from hand to zone. Concluded with a developer README covering local setup, architecture, tests, and deploy.

### Phase 14: Gameplay Zone Infrastructure

**Goal**: A personal spread zone exists for each connected player and a shared communal spread zone exists on the table — all cards visible simultaneously, not stacked
**Depends on**: Phase 13
**Requirements**: PLAY-01, PLAY-02
**Plans**: 6 plans (2 original + 4 gap-closure)

Plans:
- [x] 14-01-PLAN.md — Extend types (Pile.region/ownerId, ClientGameState.myPlayZoneId), seed communal zone, idempotent personal zone creation in onConnect, onStart migration, viewFor exposure, unit tests
- [x] 14-02-PLAN.md — SpreadZone component, BoardView 4-section restructure, opponent spread zones in header, Playwright e2e
- [x] 14-03-PLAN.md — GAP-01: DEAL_CARDS late-joiner hand initialization
- [x] 14-04-PLAN.md — GAP-02/03/04: Spread drop bypass dialog, sortable cards, convert play pile to spread communal
- [x] 14-05-PLAN.md — GAP-05: HandZone -ml-5 cascade overlap
- [x] 14-06-PLAN.md — GAP-06: Intra-spread reorder inserts at correct index

**Details:**
Spread zones implemented as `Pile` records with `region: "spread"` and optional `ownerId`. No new collection type — `MOVE_CARD`, `UNDO_MOVE`, and `RESET_TABLE` work with spread zone IDs unchanged. `onConnect` creates `spread-{playerToken}` pile idempotently. The existing `play` pile was converted to `region: "spread"` communal zone; `spread-communal` was removed via onStart migration with card transfer. `SpreadZone.tsx` renders cascade with `SortableContext + useSortable` per face-up card, `useDndMonitor` for intra-zone reorder dispatching `REORDER_PILE_SPREAD`. BoardView restructured into 4 sections: opponent header / table / spread row / hand. 130 tests passing.

6 gaps found and closed during execution: late-joiner hand init, spread drop dialog bypass, sortable spread cards, play-pile-as-communal migration, HandZone -ml-5 cascade, intra-spread reorder insert position. VERIFICATION.md status: `human_needed` — 5 behaviors require live 2-player session.

Key decisions:
- Spread zones as Pile records (not a new type) — all existing message handlers work unchanged
- `isIntraSpreadReorder` guard in BoardDragLayer skips MOVE_CARD for same-pile reorders; SpreadZone's REORDER_PILE_SPREAD fires uncontested
- `aria-pressed` placed after `{...attributes}` spread — dnd-kit's attributes object includes its own aria-pressed; explicit override must come last

**Goal**: Player can select 1–5 cards from hand and play them as a set into their personal zone or the communal zone in one atomic action
**Depends on**: Phase 14
**Requirements**: PLAY-03
**Plans**: 3 plans

Plans:

**Details:**

Selection UX: `selectedIds` state lifted to `BoardDragLayer`, click-to-toggle on `SortableHandCard` (ring-1/primary/30 + 6px translateY lift), "N selected" badge in `HandZone` label, Escape + pointer-down-outside clearing. `isMultiCardSet` guard in `handleDragEnd` dispatches `PLAY_CARD_SET` with all selectedIds when dragging from selection. Phase 15 Plan 03 fixed the stale `spread-zone-spread-communal` testid from Phase 14 to `spread-zone-play`. VALIDATION.md: `nyquist_compliant: true`. UAT: 7/8 (1 cosmetic resolved — ring softened).

Key decisions:
- `aria-pressed` after `{...attributes}` spread (same pattern as Phase 14) — TS2783 otherwise
- `:not(:has([role="button"]))` to count leaf card elements in spread zone — each card renders two nested [role="button"] divs; leaf selector gives correct count
- Softened ring from `ring-2 ring-primary` to `ring-1 ring-primary/30` per UAT feedback
