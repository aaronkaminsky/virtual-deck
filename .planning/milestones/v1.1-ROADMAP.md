# Milestone v1.1: Social Identity + UX Polish

**Status:** ✅ SHIPPED 2026-04-19
**Phases:** 999.10, 999.11 (pre-work) + 9, 10, 11
**Total Plans:** 11

## Overview

Social Identity + UX Polish adds player identity and presence to the game, making the table feel like a real shared space where everyone knows who they're sitting with. The milestone also closes drag-and-drop UX gaps: drop zones now match visual boundaries, pile position dialog is keyboard-navigable, empty piles skip the dialog entirely, and every deal is automatically shuffled first for fairness.

### Phase 10: Shuffle Before Deal

**Goal**: The deal action is always fair — dealing from any pile automatically shuffles it first so players cannot predict card order
**Depends on**: Phase 9
**Requirements**: GAME-01
**Plans**: 3/3 complete

Plans:
- [x] 10-01-PLAN.md — Add PILE_SHUFFLED event type and DEAL_CARDS server surgery (shuffle + broadcast + 650ms delay)
- [x] 10-02-PLAN.md — Wire PILE_SHUFFLED broadcast into SHUFFLE_PILE handler
- [x] 10-03-PLAN.md — Client card-fan animation in PileZone driven by PILE_SHUFFLED event

**Details:**
`DEAL_CARDS` handler now shuffles the pile first using the existing Fisher-Yates + crypto.getRandomValues logic, then broadcasts `PILE_SHUFFLED` before a 650ms delay. Client-side `usePartySocket` tracks `shufflingPileIds: Set<string>` — IDs enter on `PILE_SHUFFLED` and exit after 650ms. `PileZone` renders 5 ghost card divs with staggered `pile-fan-spread` CSS keyframe animation when its pile ID is in the shuffling set.

## Milestone Summary

**Pre-work Phases:** 999.10 (drag origin placeholder) and 999.11 (pile dialog keyboard UX) were inserted before the v1.1 milestone proper. They addressed UX polish found during v1.0 playtesting.

**Key Decisions:**
- `displayName` required string (not optional) on Player — consistent presence; empty string default avoids null checks everywhere
- Deferred WebSocket connect via `enabled` flag — name must be set before connecting so `?name=` param is available on handshake
- `joinState` null-check gates both socket and board render — single source of truth for join progression; avoids partial render before connection
- `pointerWithin` custom collision detection — drops only register when pointer is physically inside target rect, matching visual element boundaries

**Issues Resolved:**
- Drop zones matching visual boundaries (Phase 999.10): fixed via `pointerWithin` collision detection
- Pile dialog no keyboard cancel path (Phase 999.11): fixed via `onOpenChange` cancel branch + `initialFocus`
- Players anonymous at table: fixed via Phase 9 identity system
- Deal fairness (card order predictable from pile state): fixed via auto-shuffle in Phase 10

**Technical Debt Incurred:**
- Phase 10 `getConnections()` iterator is cached once per handler call (WR-01 fix); if PartyKit changes the API this may need revisiting
