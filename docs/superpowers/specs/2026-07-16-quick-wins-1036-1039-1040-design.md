# Design: Quick-wins batch — 1036, 1039, 1040

**Date:** 2026-07-16
**Backlog items:** 1036 (mask LAST_MOVE card ids for hand destinations), 1039 (face up/down control visible on empty tableau), 1040 (rick roll plays longer to reach lyrics)

Three small, independent fixes shipped on one branch. No shared code between them; each is a localized change with its own test.

## 1040 — Rick roll timer

**Problem:** The rickroll overlay auto-dismisses after 10 seconds, but the lyrics start ~18 seconds into the video, so it never reaches them.

**Change:** In `src/components/RickrollOverlay.tsx`, change the auto-dismiss timeout from `10_000` to `30_000` (~12 seconds of lyric time). The intro riff is kept; the manual ✕ dismiss button continues to work throughout.

**Decision:** Extend the timer rather than adding `&start=17` to the embed URL — keeping the iconic intro was preferred. Note the embed autoplays muted (browser autoplay policy), so "reaching the lyrics" is visual unless the user interacts with the player.

**Tests:** `tests/rickrollOverlay.test.ts` pins the timeout via source regex (`setTimeout(() => setRun(0), 10_000)`). Update the regex to expect `30_000` first (RED), then change the component (GREEN). The e2e in `playwright/easterEggs.spec.ts` only asserts visibility and manual dismiss — unaffected.

## 1039 — Face toggle visible on empty tableau

**Problem:** The face up/down control (Eye/EyeOff) on a spread zone disappears when the zone is empty, so a player cannot set landing orientation before dropping cards.

**Change:** In `src/components/SpreadZone.tsx`, the controls row is gated by `interactive !== false && !isReallyEmpty`. Drop the `!isReallyEmpty` condition so the full controls row (face toggle + select-all, plus chip controls when chips are enabled) renders for the empty zone too.

**Placement:** When the zone is empty, the controls row floats above the panel (`absolute bottom-full left-0`, in the rail column's empty bottom space) so it costs no flow height — the board's `min-h-[560px]` floor is unchanged. Because the floating row overhangs content outside its zone, it is pointer-events-gated while hidden (`.zone-controls-floating` in `globals.css`, same rationale as `.canvas-pile-controls`); hovering or focusing the zone reveals it. Populated zones keep the existing in-flow controls row below the strip.

**Why the whole row (not just the face toggle):** Select-all already disables itself at zero cards, and the chip "Pot" button already disables at zero chips — no new guards needed, and the layout stays consistent whether the zone is empty or not. `SET_PILE_FACE` works on an empty pile server-side. The row keeps its existing hover-reveal behavior (`zone-controls` CSS).

**Verification note:** The empty tableau placeholder is a slim dashed strip (`h-4`/`h-5`); visually confirm the floating controls land in the rail column's empty space without shifting surrounding layout, and that the board still fits the declared 560px minimum viewport.

**Tests:** No component render harness exists (unit tests are source-regex or server-logic style), so: a Playwright assertion that the face toggle is visible and clickable on an empty tableau, plus a source-regex unit test pinning that the controls-row gate no longer includes `!isReallyEmpty`.

## 1036 — Mask LAST_MOVE card ids for hand destinations

**Problem:** `broadcastLastMove` in `party/index.ts` sends the moved card ids to **every** connection, including when the destination is a private hand — a pre-existing information leak noted in the 1031 final review.

**Change:** In `broadcastLastMove`, when `toZoneType === "hand"`, send the real `cardIds` only to the connection whose `getPlayerToken(conn)` equals `toZoneId` (hands are keyed by player token, and `Player.id` *is* the token). All other connections receive `cardIds: []`.

**Why this is safe with no client change:**
- `OpponentHand.tsx` renders its highlight from `toZoneType`/`toZoneId` alone — it never reads `cardIds` — so opponents still see the "cards landed in this hand" pulse.
- The receiving player's own `HandZone` uses `cardIds` for per-card highlighting and still receives real ids.

**Invariant:** Masking applies even when the destination hand is revealed. Simpler rule — hand-destination card ids never leave the owner's connection — and opponents lose nothing since their highlight ignores ids.

**Tests:** Server unit test with two connections: move a card into player A's hand; assert A's `LAST_MOVE` event contains the card id and B's contains `[]`. Also assert pile/canvas destinations still broadcast real ids to everyone.
